Executive answer

If the question is "Is third_party/ supposed to be permanent?", the answer is no. The authoritative plan docs treat it as temporary donor/reference code.

If the question is "Can we delete it today without breaking the current app?", probably yes. The app is already self-contained at runtime.

If the question is "Can we delete it while still claiming we are following the current plan?", no. The current plans still assume unfinished donor-port work from agent-flow and sidecar.

Runtime answer: safe to remove.

Decision options

  • Keep third_party/ for now: best option if the current plan docs remain authoritative.
  • Remove it now and rewrite the plan downward: valid only if you are intentionally abandoning the unfinished donor-port scope.
  • Middle path: keep only until the active branch work is either merged or formally rejected, then replace the raw checkouts with distilled research notes.
Recommended: treat third_party/ as temporary donor code exactly as intended, but do not delete it until the current plan is either finished or explicitly narrowed.

Bottom line

The confusion is not really about what third_party/ is for. That part is straightforward: it is temporary donor code. The real confusion is that the repo is split between a mostly healthy Phase 1 prototype on main, a set of partially built feature branches, and plan docs that still assume those donor ports are part of the official finish line.

So the honest answer is:

  • third_party/ has served its purpose for runtime independence already.
  • third_party/ has not yet served its purpose for plan completion.
  • If the plan stays the plan, keep it.
  • If the plan changes, delete it and update the docs so the repo stops pretending those donor ports are still mandatory.
