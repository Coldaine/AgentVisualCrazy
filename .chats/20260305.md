## codex: $find-skills ## ️ Polycam Raw Data Structure

$find-skills ## ️ Polycam Raw Data Structure

From the polyform spec, the export looks like this:

```
capture-folder/
  raw.glb              # reconstructed mesh (binary glTF)
  thumbnail.jpg
  polycam.mp4
  mesh_info.json       # metadata + alignmentTransform
  keyframes/
    images/            # RGB keyframes
    corrected_images/  # post-optimization RGB
    cameras/           # raw ARKit camera JSON (pose + intrinsics)
    corrected_cameras/ # optimized camera JSON
    depth/             # 16-bit PNG depth maps
    confidence/        # confidence maps per frame
```

##  Best 3D File Type for Comparison

**PLY.** It's the lingua franca across your whole toolchain: RTAB-Map exports it, Polycam exports it, Open3D reads it, CloudCompare reads it, Nerfstudio reads it. It carries both geometry and color, works as point cloud or mesh, and it's unambiguous. OBJ is fine for textured meshes but PLY is simpler for programmatic comparison (computing distances, overlap, registration error).

Export Polycam's processed result as **PLY point cloud** for the apples-to-apples comparison.

# 12-Step Piloting Plan: Driving Azur Lane via scrcpy-mcp

**Goal:** Prove the agent can pilot the Azur Lane emulator purely via vision and existing tooling (`scrcpy-mcp`), without relying on brittle UI hierarchies or writing custom ADB wrappers.

**Prerequisites (Already Complete):**
- `scrcpy-mcp` is cloned, built, and verified. It provides 34 MCP tools including `screenshot`, `tap`, `swipe`, and `app_start`.
- MEmu emulator is running and connected via ADB at `127.0.0.1:21513`.
- `gemini_flash.py` (VLM capability) is available for coordinate extraction if needed.

## The 12 Steps

1. **Configure Environment:** Set `ADB_PATH` to the MEmu specific `adb.exe` and `ANDROID_SERIAL` to `127.0.0.1:21513` to ensure `scrcpy-mcp` targets the correct emulator.
2. **Initial Vision Check (`screenshot`):** Call the `scrcpy-mcp` `screenshot` tool. Retrieve the PNG and view it to understand the current state of the emulator screen.
3. **App Verification (`app_current` / `app_start`):** Call `app_current` to confirm Azur Lane is the foreground app. If not, use `app_start` (with the package name `com.YoStarEN.AzurLane`) to launch it.
4. **Interactive Piloting (`tap`):** Based on the visual output from step 2, determine the necessary action (e.g., closing a popup, starting the game). Call the `scrcpy-mcp` `tap` tool with the appropriate `x` and `y` coordinates.
5. **Verify Action (`screenshot`):** Immediately call the `screenshot` tool again to verify that the tap registered and the UI transitioned as expected.
6. **Navigation - Main Menu:** Repeat the `screenshot` -> `tap` -> `verify` loop to navigate through any login screens, daily reward popups, or announcements until the primary **Main Menu** (Lobby) is reached and confirmed visually.
7. **Navigation - Commissions:** From the Main Menu, locate the Commissions button visually. Execute a `tap` to enter the Commissions screen.
8. **Verify Commissions:** Take a `screenshot` to confirm the UI has transitioned to the Commissions screen.
9. **Navigation - Return to Main:** Locate the 'Back' or 'Home' button visually. Execute a `tap` (or use the `keyevent` tool for BACK: `4`) to return to the Main Menu. Verify with a `screenshot`.
10. **Navigation - Dorm:** From the Main Menu, locate the Dorm button visually. Execute a `tap` to enter the Dorm screen. Verify with a `screenshot`.
11. **Navigation - Return to Main:** Execute a `tap` on the 'Back' or 'Home' button to return to the Main Menu. Verify with a `screenshot`.
12. **Record Blueprint:** Compile the sequence of (screenshot_before, action, screenshot_after) for the successful navigation paths into a structured JSONL log. This serves as the manual "blueprint" that proves the flow works and can be codified into deterministic tools.

**Execution Rule:** Zero new plumbing code. All actions are performed by invoking the existing tools provided by `scrcpy-mcp` (run via `npx @modelcontextprotocol/inspector node dist/index.js` or standard MCP integration). The agent relies on its own visual processing capabilities to determine coordinates and verify success, falling back to Gemini Flash only if semantic reasoning is required to find an element.
