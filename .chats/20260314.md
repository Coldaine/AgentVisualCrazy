## Phase 2: Signal Prospecting — What's Actually Useful?

**Goal:** Test each metadata signal for whether it reliably distinguishes
"deep thinking" from "operational chatter."

### 2a. formatting_divergence_score distribution

```sql
-- Distribution: is this a useful signal?
SELECT
  CASE
    WHEN formatting_divergence_score IS NULL THEN 'null'
    WHEN formatting_divergence_score < 0.1 THEN '0.0-0.1'
    WHEN formatting_divergence_score < 0.3 THEN '0.1-0.3'
    WHEN formatting_divergence_score < 0.5 THEN '0.3-0.5'
    WHEN formatting_divergence_score < 0.7 THEN '0.5-0.7'
    ELSE '0.7+'
  END as bucket,
  count(*),
  avg(array_length(string_to_array(chosen_text, ' '), 1)) as avg_words
FROM wispr_corpus_entries
GROUP BY bucket
ORDER BY bucket;
```

Then pull 10 rows from the TOP of the distribution and 10 from the BOTTOM.
Read them. Does high divergence actually mean complex/abstract thinking?
Or does it mean something else (transcription errors, code dictation, etc.)?

### 2b. Cognitive friction (duration - speech_duration)

```sql
-- Same approach: look at the distribution
SELECT
  CASE
    WHEN (duration - speech_duration) IS NULL THEN 'null'
    WHEN (duration - speech_duration) < 1 THEN 'under 1s'
    WHEN (duration - speech_duration) < 5 THEN '1-5s'
    WHEN (duration - speech_duration) < 15 THEN '5-15s'
    ELSE '15s+'
  END as pause_bucket,
  count(*),
  avg(array_length(string_to_array(chosen_text, ' '), 1)) as avg_words
FROM wispr_corpus_entries
GROUP BY pause_bucket
ORDER BY pause_bucket;
```

Pull and read examples from each bucket. Does heavy pausing mean deep thought?
Or does it mean distraction, multitasking, interruptions?

### 2c. App context as signal

```sql
-- What apps does he dictate into?
SELECT app_name, count(*), 
       avg(array_length(string_to_array(chosen_text, ' '), 1)) as avg_words
FROM wispr_corpus_entries
GROUP BY app_name
ORDER BY count(*) DESC;
```

Are certain apps strongly correlated with substantive thinking?
(Claude conversations probably yes. Terminal probably no. But CHECK.)

### 2d. ASR confidence as signal

```sql
-- Does average_log_prob correlate with anything useful?
SELECT
  CASE
    WHEN average_log_prob IS NULL THEN 'null'
    WHEN average_log_prob > -0.3 THEN 'high confidence'
    WHEN average_log_prob > -0.6 THEN 'medium'
    ELSE 'low confidence'
  END as confidence,
  count(*),
  avg(formatting_divergence_score) as avg_divergence,
  avg(array_length(string_to_array(chosen_text, ' '), 1)) as avg_words
FROM wispr_corpus_entries
GROUP BY confidence;
```

Low ASR confidence might mean unusual vocabulary (technical terms, novel
framings) which could actually be a POSITIVE signal for idea detection.
Or it might mean mumbling. Look at examples.

### 2e. Text length as signal

```sql
-- Long utterances: are they ideas or rambling?

WHERE chosen_text IS NOT NULL

```

### 2f. What's in conversation_id?

## Phase 2: Signal Prospecting — What's Actually Useful?

**Goal:** Test each metadata signal for whether it reliably distinguishes

### 2a. formatting_divergence_score distribution

```sql
-- Distribution: is this a useful signal?
SELECT
  CASE
    WHEN formatting_divergence_score IS NULL THEN 'null'
    WHEN formatting_divergence_score < 0.1 THEN '0.0-0.1'
    WHEN formatting_divergence_score < 0.3 THEN '0.1-0.3'
    WHEN formatting_divergence_score < 0.5 THEN '0.3-0.5'
    WHEN formatting_divergence_score < 0.7 THEN '0.5-0.7'
    ELSE '0.7+'
  END as bucket,
  count(*),
  avg(array_length(string_to_array(chosen_text, ' '), 1)) as avg_words
FROM wispr_corpus_entries
GROUP BY bucket
ORDER BY bucket;
```

Then pull 10 rows from the TOP of the distribution and 10 from the BOTTOM.

### 2b. Cognitive friction (duration - speech_duration)

```sql
-- Same approach: look at the distribution
SELECT
  CASE
    WHEN (duration - speech_duration) IS NULL THEN 'null'
    WHEN (duration - speech_duration) < 1 THEN 'under 1s'
    WHEN (duration - speech_duration) < 5 THEN '1-5s'
    WHEN (duration - speech_duration) < 15 THEN '5-15s'
    ELSE '15s+'
  END as pause_bucket,
  count(*),
  avg(array_length(string_to_array(chosen_text, ' '), 1)) as avg_words
FROM wispr_corpus_entries
GROUP BY pause_bucket
```

# Idea Emergence Visualizer: Corpus Exploration Playbook

## Purpose

This is an iterative exploration protocol for an agent with direct access to
Patrick's speech-to-text corpus in PostgreSQL. The goal is to understand what's
actually in the data, discover what signals are reliable, find where the gaps
are, and figure out how to extract and reconnect ideas from isolated fragments.

DO NOT prescribe a pipeline. DISCOVER one by looking at the data.

---

## Ground Rules for the Agent

1. Every phase starts with LOOKING AT ACTUAL DATA before forming hypotheses.
   Print real rows. Read real text. Count real things. No assumptions.

2. After each phase, write a short findings memo: what you found, what
   surprised you, what you now think is true, and what you want to check next.

3. Patrick will review findings memos and redirect. The playbook is a starting
   structure, not a rigid script. Follow the data.

4. Keep a running "signal inventory": which columns/features are actually
   useful for what, which ones looked promising but aren't, and which
   combinations work together.

## Phase 1: Orientation — What Am I Looking At?

**Goal:** Build a mental model of the corpus. Not statistics about it. A
feel for what's in it.

### 1a. Schema and scale

```sql
-- What tables exist? What are the columns?
\d+ wispr_corpus_entries

-- (If other tables exist, list and describe them too)

-- How many total rows?
SELECT count(*) FROM wispr_corpus_entries;

-- Date range?
SELECT min(created_at), max(created_at) FROM wispr_corpus_entries;

-- Average utterance length in words?
SELECT avg(array_length(string_to_array(chosen_text, ' '), 1))
FROM wispr_corpus_entries
WHERE chosen_text IS NOT NULL;
```

### 1b. Read 50 random rows, end to end

Don't summarize. Don't filter. Just read them. Get a feel for what a
typical row looks like. Note:

- How long are utterances? One sentence? A paragraph? A word?
- How much is operational ("open the file", "run the build") vs
  substantive thinking?
- What does the app_name distribution look like?
- What's in ax_text and textbox_contents? Is it always populated?
- How often is chosen_text different from asr_text?

```sql
SELECT id, created_at, app_name, asr_text, chosen_text,
       ax_text, textbox_contents, url,
       formatting_divergence_score,
       duration, speech_duration
FROM wispr_corpus_entries
ORDER BY random()
LIMIT 50;
```

### 1c. Read 50 CONSECUTIVE rows from a random day

This is different from random sampling. You want to see what a "stream"
of Patrick's dictation looks like in sequence. Does it feel like a
narrative? Does he switch topics every utterance? Are there runs of
related dictation?

```sql
-- Pick a random day
SELECT DISTINCT date(created_at) as day
FROM wispr_corpus_entries
ORDER BY random() LIMIT 1;

-- Then pull 50 consecutive rows from that day
SELECT id, created_at, app_name, asr_text, chosen_text,
       formatting_divergence_score, duration, speech_duration
FROM wispr_corpus_entries
WHERE date(created_at) = '[THE DAY YOU PICKED]'
ORDER BY created_at
LIMIT 50;
```

Read these in order. What do you notice about adjacency? About app switching?
About topic continuity or lack thereof?

### 1d. Findings memo #1

Write up:
- What the corpus actually feels like (not what you expected)

- Whether consecutive utterances seem related or unrelated
- What metadata fields are reliably populated vs sparse

## Phase 2: Signal Prospecting — What's Actually Useful?

**Goal:** Test each metadata signal for whether it reliably distinguishes
"deep thinking" from "operational chatter."

### 2a. formatting_divergence_score distribution

```sql
-- Distribution: is this a useful signal?
SELECT
  CASE
    WHEN formatting_divergence_score IS NULL THEN 'null'
    WHEN formatting_divergence_score < 0.1 THEN '0.0-0.1'
    WHEN formatting_divergence_score < 0.3 THEN '0.1-0.3'
    WHEN formatting_divergence_score < 0.5 THEN '0.3-0.5'
    WHEN formatting_divergence_score < 0.7 THEN '0.5-0.7'
    ELSE '0.7+'
  END as bucket,
  count(*),
  avg(array_length(string_to_array(chosen_text, ' '), 1)) as avg_words
FROM wispr_corpus_entries
GROUP BY bucket
ORDER BY bucket;
```

Then pull 10 rows from the TOP of the distribution and 10 from the BOTTOM.
Read them. Does high divergence actually mean complex/abstract thinking?
Or does it mean something else (transcription errors, code dictation, etc.)?

### 2b. Cognitive friction (duration - speech_duration)

```sql
-- Same approach: look at the distribution
SELECT
  CASE
    WHEN (duration - speech_duration) IS NULL THEN 'null'
    WHEN (duration - speech_duration) < 1 THEN 'under 1s'
    WHEN (duration - speech_duration) < 5 THEN '1-5s'
    WHEN (duration - speech_duration) < 15 THEN '5-15s'
    ELSE '15s+'
  END as pause_bucket,
  count(*),
  avg(array_length(string_to_array(chosen_text, ' '), 1)) as avg_words
FROM wispr_corpus_entries
GROUP BY pause_bucket
ORDER BY pause_bucket;
```

Pull and read examples from each bucket. Does heavy pausing mean deep thought?
Or does it mean distraction, multitasking, interruptions?

### 2c. App context as signal

```sql
-- What apps does he dictate into?
SELECT app_name, count(*), 
       avg(array_length(string_to_array(chosen_text, ' '), 1)) as avg_words
FROM wispr_corpus_entries
GROUP BY app_name
ORDER BY count(*) DESC;
```

Are certain apps strongly correlated with substantive thinking?
(Claude conversations probably yes. Terminal probably no. But CHECK.)

### 2d. ASR confidence as signal

```sql
-- Does average_log_prob correlate with anything useful?
SELECT
  CASE
    WHEN average_log_prob IS NULL THEN 'null'
    WHEN average_log_prob > -0.3 THEN 'high confidence'
    WHEN average_log_prob > -0.6 THEN 'medium'
    ELSE 'low confidence'
  END as confidence,
  count(*),
  avg(formatting_divergence_score) as avg_divergence,
  avg(array_length(string_to_array(chosen_text, ' '), 1)) as avg_words
FROM wispr_corpus_entries
GROUP BY confidence;
```

Low ASR confidence might mean unusual vocabulary (technical terms, novel
framings) which could actually be a POSITIVE signal for idea detection.
Or it might mean mumbling. Look at examples.

### 2e. Text length as signal

```sql
-- Long utterances: are they ideas or rambling?
SELECT id, created_at, app_name,
       array_length(string_to_array(chosen_text, ' '), 1) as word_count,
       left(chosen_text, 200) as preview
FROM wispr_corpus_entries
WHERE chosen_text IS NOT NULL
ORDER BY array_length(string_to_array(chosen_text, ' '), 1) DESC
LIMIT 20;

-- Short utterances: are they always noise?
SELECT id, created_at, app_name, chosen_text
FROM wispr_corpus_entries
WHERE chosen_text IS NOT NULL
  AND array_length(string_to_array(chosen_text, ' '), 1) BETWEEN 3 AND 10
ORDER BY random()
LIMIT 30;
```

### 2f. What's in conversation_id?

```sql
-- How many distinct conversation_ids? How many rows per conversation?
SELECT count(distinct conversation_id),
       avg(cnt), min(cnt), max(cnt)
FROM (
  SELECT conversation_id, count(*) as cnt
  FROM wispr_corpus_entries
  WHERE conversation_id IS NOT NULL
  GROUP BY conversation_id
) sub;
```

Does conversation_id actually group related utterances? Or is it a Wispr Flow
session ID that just means "these were dictated close in time"? Pull a few
conversation groups and read them to find out.

### 2g. ax_text and textbox_contents

```sql
-- How often are these populated?
SELECT
  count(*) as total,
  count(ax_text) as has_ax_text,
  count(textbox_contents) as has_textbox
FROM wispr_corpus_entries;

-- When they ARE populated, what's in them?
SELECT created_at, app_name, left(chosen_text, 100) as utterance,
       left(ax_text, 200) as ax,
       left(textbox_contents, 200) as textbox
FROM wispr_corpus_entries
WHERE ax_text IS NOT NULL
ORDER BY random()
LIMIT 20;
```

ax_text might contain what was on screen. textbox_contents might contain
what was already typed in the input field before dictation. These could
be critical context signals, or they could be noisy/unreliable. LOOK.

### 2a. formatting_divergence_score distribution

### 2b. Cognitive friction (duration - speech_duration)

Or does it mean distraction, multitasking, interruptions?

- Was it keyword overlap? Semantic similarity? Both?
- Were there hits that LOOKED related but actually weren't?
- Were there gaps (weeks with no hits) where the idea presumably
  still existed in Patrick's head but wasn't dictated?

### 4b. Test temporal adjacency

For the hits you found: look at what was dictated BEFORE and AFTER each hit
(within 5-10 minutes). Is there contextual material around the idea mention
that helps? Or is each mention completely isolated?

```sql
-- For a specific hit, pull surrounding utterances
SELECT created_at, app_name, left(chosen_text, 150) as text,
       formatting_divergence_score
FROM wispr_corpus_entries
WHERE created_at BETWEEN '[HIT_TIMESTAMP]'::timestamp - interval '10 minutes'
                     AND '[HIT_TIMESTAMP]'::timestamp + interval '10 minutes'
ORDER BY created_at;
```

### 4c. Test app-context adjacency

When Patrick dictates an idea fragment while looking at Claude (app_name),
and there's textbox_contents or ax_text, does that context help explain
what the fragment is about?

- What's the false positive rate for each linking approach?
- How dense or sparse are idea threads? (many mentions or few?)
- Recommendation: how should the extraction pipeline actually work,

1. Partition first by `source_device` and normalized `app` family.
2. Filter for candidate idea-rich material using entry shape and context:

3. Start thread discovery from:
   - rare lexical anchors
   - manually supplied seed phrases
   - semantic seeds for known ideas
4. Expand each seed with:
   - semantic nearest neighbors
   - lexical variants and paraphrase probes
   - local plus/minus time windows
5. Accept a thread only if conceptual continuity survives all three:
   - lexical or semantic relation
   - local context coherence
   - app or work-mode plausibility
6. Label each thread state as one of:
   - exact articulation
   - paraphrase articulation
   - adjacent evidence
   - absent in Wispr
