## codex: Yes. Here’s a **single agent prompt** optimized fo...

Yes. Here’s a **single agent prompt** optimized for **“something fun working in 30 minutes”** rather than perfection. The goal is not elegant infrastructure. The goal is a working toy with a browser UI and an API-backed local video model on your 5090. Little chaos goblin, high yield. ⚙️

Use this as-is:

```text
You are setting up a local LAN-accessible text-to-video demo on a Windows machine with an RTX 5090, using WSL2 Ubuntu, with the explicit goal of having something entertaining and usable by children within 30 minutes.

Your mission is speed to a working demo, not architectural purity.

Success criteria:
1. A browser frontend exists and is usable from another laptop on the same LAN.
2. A user can type a silly prompt, click generate, and receive a short video clip.
3. The system runs locally on the 5090 inside WSL2.
4. The demo is reasonably robust for light family use.
5. The entire setup is simple enough that a human can restart it manually if needed.

Hard constraints:
- Prioritize the fastest path to a working result.
- Do not overengineer.
- Do not spend time comparing many model options once a viable choice is identified.
- Do not build a complex auth system, database, job framework, or cloud deployment.
- Do not get distracted by theoretical improvements.
- Prefer battle-tested open-source tooling with an existing UI and API.
- Bind services so they can be reached from the LAN, not only localhost.
- The frontend must be simple and kid-friendly.
- The generated clips can be short and lowish resolution if needed.
- Default to safe, fun prompts and a short queue.

Primary implementation strategy:
- Use ComfyUI as the backend because it already has a UI, workflow system, and API.
- Use the fastest practical open video model that can run locally on a 5090 in WSL with minimal setup friction.
- Prefer an existing ComfyUI workflow for text-to-video or image-to-video rather than building inference code from scratch.
- If model setup becomes too slow or unstable, fall back immediately to the next simpler viable model or workflow.
- Build a very small frontend that talks to a tiny wrapper API, or use ComfyUI directly if that is faster and can be made kid-friendly enough.
- If a custom frontend is required, make it a single static HTML page with a prompt box, a few preset buttons, a generate button, progress text, and a video result area.
- The frontend should include preset fun prompts like:
  - “A baby dragon making pancakes”
  - “A pirate corgi surfing on soup”
  - “A tiny wizard riding a raccoon through space”
  - “A dinosaur having a birthday party”
- Keep generation defaults conservative:
  - short clips
  - modest resolution
  - one job at a time
  - simple queue handling

Environment assumptions:
- Host OS: Windows
- Compute: RTX 5090
- Runtime: WSL2 Ubuntu already available or installable
- Access: another laptop on the same local network
- Existing frontend may exist, but if adapting it slows things down, create a minimal new frontend instead

Networking requirements:
- Ensure the service is reachable from the laptop over LAN.
- Use 0.0.0.0 bindings where appropriate.
- Handle WSL networking realities pragmatically.
- If needed, configure Windows port forwarding or equivalent, but only if necessary to make the demo reachable.

Implementation order:
1. Verify GPU visibility in WSL.
2. Install the minimal Python/CUDA/PyTorch stack required for the chosen backend.
3. Install and launch ComfyUI.
4. Install the chosen video model/workflow with the least friction.
5. Prove a single local generation works.
6. Expose the service/API to the LAN.
7. Build or adapt a tiny frontend.
8. Test end-to-end from the laptop browser.
9. Leave behind dead-simple run instructions.

Decision rules:
- Every 5–10 minutes, reassess whether the current path is still the fastest path to a working demo.
- If blocked on model size, dependencies, or workflow instability, pivot fast.
- A lower-quality working demo is better than a higher-quality half-built system.
- Do not disappear into configuration rabbit holes.

Deliverables:
1. A working local service with clear commands to start it.
2. A browser frontend usable from the laptop.
3. A small README with:
   - what was installed
   - how to run it
   - where to open it from the laptop
   - known limitations
4. A short summary of what model/workflow was chosen and why.
5. A list of any shortcuts taken due to the 30-minute time constraint.

Technical preference guidance:
- Strongly prefer ComfyUI plus an existing workflow/API.
- Prefer a model family known to run on consumer GPUs rather than giant research checkpoints that are likely to stall setup.
- Prefer a thin wrapper over writing custom orchestration.
- Prefer static files and tiny FastAPI/Node servers over heavy frameworks.
- Prefer polling over elaborate websocket engineering if polling is faster to implement.

UX guidance for the kid-facing frontend:
- Big buttons
- Minimal text
- Friendly labels
- A few one-click example prompts
- “Generating…” status
- Embedded playable output
- A “Try another silly idea” feel
- No admin-looking controls

Important:
- Keep a running log of major decisions and commands used.
- At the end, provide exact run commands and the LAN URL(s).
- If something cannot be completed, degrade gracefully and leave the closest possible working demo rather than stopping.

Your output should be action-oriented and implementation-driven.
You are not writing an essay.
You are building the demo.
```
