This document is the borrowed-tool intake note for the emulator path.

It is not the canonical runtime architecture plan.

For the runtime blueprint, use:
- [multi-tier-runtime-implementation-plan-2026-03-24.md](/mnt/d/_projects/MasterStateMachine/docs/plans/multi-tier-runtime-implementation-plan-2026-03-24.md)

## Purpose

This note answers only these questions:

- which borrowed tools should be used for MEmu 9 attachment and interaction
- what baseline integration is needed before higher-level runtime work
- which emulator-specific constraints shape that path

It does not own tiering policy, semantic cache design, teacher escalation, or runtime architecture sequencing.

## Tool Posture

The borrowed substrate posture is:

- `MaaMCP` is the primary agent-facing control tool for connect, screenshot, tap, swipe, key, text, and health/status if available
- `scrcpy` is the preferred live visual substrate and operator/debug stream
- `adbfriend` is installed and documented separately for your own use, but it is not part of the runtime path

The runtime should borrow these tools rather than reimplementing attachment, capture, and input plumbing.

## What This Layer Owns

- device connection and basic health checks
- screenshot/frame access from the emulator path
- action dispatch primitives
- emulator-specific bootstrap and recovery commands
- evidence needed to prove a frame/action loop is reliable enough for runtime work above it

## What This Layer Does Not Own

- VLM task semantics
- semantic cache policy
- hit/hint/miss routing
- teacher escalation
- workflow reasoning

Those belong to the canonical runtime plan, not to this integration note.

## Unity Constraint

The target game is Unity-first.

That means structured Android UI hierarchy tooling should not be treated as the semantic layer for the runtime. In particular:

- `uiautomator2` may still be useful for diagnostics or limited Android shell interaction
- it should not be treated as the core observation or semantics path for the live Unity workflow

The runtime must be able to operate from borrowed control-tool frames and action verification rather than assuming a rich accessibility tree.

## Practical Integration Shape

The narrow integration loop is:

1. connect to the MEmu instance through the borrowed control tool
2. attach the preferred visual substrate for observation and operator/debug visibility
3. dispatch primitive actions through the borrowed control tool
4. capture again
5. verify that the loop is stable enough for the runtime layer above it

This note intentionally stops there.

## Deferred Or Optional Tools

- `scrcpy`
  - useful as the preferred visual/debug substrate
  - not the semantic owner of the runtime
- `uiautomator2`
  - optional diagnostic/helper tool
  - not the primary semantics path for Unity interaction
- `DroidMind` and similar orchestration layers
  - deferred until a concrete transport or runtime gap requires them

## Success Gate

This note is satisfied when the emulator path can repeatedly:

- attach to the intended MEmu instance
- capture usable current frames
- dispatch actions
- verify that the loop is stable on one real session

Once that gate is met, the canonical runtime plan owns the next architectural step.

## What actually worked

From the log:

  * `adb_reachable: true`

So the transport layer was **never fundamentally failing to reach MEmu**.

## Why it looked like it couldnt connect

There are really **3 separate issues**:

### 1. The health check defined connected too strictly

`doctor` treated the absence of `maamcp` / `maafw` as a failure, even though:

* ADB could reach the emulator
* the configured serial was correct
* the device was online
* the fallback controller path worked

That means the agent was using a **tool discovery failure** as if it were a **transport connectivity failure**.

That is the main reason the session looked broken.

### 2. The session probe gated on the wrong signal

The session probe had logic like:

* run `doctor`
* if `doctor != PASS`, abort before running deeper probes

But `doctor` was failing for native tool not installed, not cannot talk to emulator.

So the probe was effectively saying:

> I refuse to test the working fallback path because the preferred path isnt present.

That is a **bad gate design**. Once the agent relaxed that gate and allowed continuation when `device_online == true`, the session probe passed.

### 3. Blank screenshots were misread as a connection symptom

The captured PNGs were valid:

* full resolution
* valid dimensions
* fast capture timing

They were just mostly white / blank.

That points to things like:

* MEmu on a blank/loading/home screen
* protected/black/empty surface content
* game not launched / wrong window state
* emulator rendering state issue

That is **not** evidence that the agent failed to connect. It is evidence that the **observation target was visually empty**.

## Root cause, in one sentence

The agent confused **native MaaMCP stack not installed** with **cannot connect to MEmu,** and then let that false failure block the rest of the probe flow.

## Architecture mistake underneath it

The system appears to have two layers of truth:

* **Transport truth**: can I reach the emulator over ADB and interact with it?
* **Preferred substrate truth**: is the preferred MaaMCP/native stack installed?

Those should produce different verdicts.

Right now the agent effectively had:

* no `maamcp`  fail

What it should have had is something closer to:

* ADB reachable + device online + fallback control works  **transport pass**
* native Maa stack unavailable  **degraded / warning**
* scrcpy usable only for debug observation  **debug_only**

## What the correct interpretation should have been

A clean interpretation of the run would be:

* **MEmu connectivity:** pass
* **ADB transport:** pass
* **Control path:** pass via fallback
* **Capture path:** pass
* **Preferred native Maa stack:** unavailable
* **scrcpy runtime observation path:** not suitable on Windows, debug only

## Why the agent seemed confused

Because it was mixing **capability detection**, **health status**, and **runtime readiness** into one binary verdict. That leads to false negatives.

It should separate:

* **reachable**
* **operable**
* **preferred stack available**
* **fully production-ready**

Those are not the same thing.

The problem was **not cant connect to MEmu.**
The real problem was:

> the probe/health logic was over-constrained and incorrectly classified a degraded-but-working setup as failure.

Thats why once the gating logic changed, everything lit up green.

## Best concise diagnosis

**MEmu was reachable the whole time.**
The failure was in the agents **health model**, not the emulator connection:

* it required `maamcp` for a PASS,
* it aborted session testing on that strict verdict,
* and it misinterpreted blank screenshots as transport trouble.

### 1. The health check defined connected too strictly

`doctor` treated the absence of `maamcp` / `maafw` as a failure, even though:

* ADB could reach the emulator
* the configured serial was correct
* the device was online
* the fallback controller path worked

That means the agent was using a **tool discovery failure** as if it were a **transport connectivity failure**.

That is the main reason the session looked broken.

### 2. The session probe gated on the wrong signal

The session probe had logic like:

* run `doctor`
* if `doctor != PASS`, abort before running deeper probes

But `doctor` was failing for native tool not installed, not cannot talk to emulator.

So the probe was effectively saying:

> I refuse to test the working fallback path because the preferred path isnt present.

That is a **bad gate design**. Once the agent relaxed that gate and allowed continuation when `device_online == true`, the session probe passed.

### 3. Blank screenshots were misread as a connection symptom

The captured PNGs were valid:

* full resolution
* valid dimensions
* fast capture timing

They were just mostly white / blank.

That points to things like:

* MEmu on a blank/loading/home screen
* protected/black/empty surface content
* game not launched / wrong window state
* emulator rendering state issue

That is **not** evidence that the agent failed to connect. It is evidence that the **observation target was visually empty**.

## Root cause, in one sentence

The agent confused **native MaaMCP stack not installed** with **cannot connect to MEmu,** and then let that false failure block the rest of the probe flow.

## Architecture mistake underneath it

The system appears to have two layers of truth:

* **Transport truth**: can I reach the emulator over ADB and interact with it?
* **Preferred substrate truth**: is the preferred MaaMCP/native stack installed?

Those should produce different verdicts.

Right now the agent effectively had:

* no `maamcp`  fail

What it should have had is something closer to:

* ADB reachable + device online + fallback control works  **transport pass**
* native Maa stack unavailable  **degraded / warning**
* scrcpy usable only for debug observation  **debug_only**

* **MEmu connectivity:** pass
* **ADB transport:** pass
* **Control path:** pass via fallback
* **Capture path:** pass
* **Preferred native Maa stack:** unavailable
* **scrcpy runtime observation path:** not suitable on Windows, debug only

## Why the agent seemed confused

Because it was mixing **capability detection**, **health status**, and **runtime readiness** into one binary verdict. That leads to false negatives.

It should separate:

* **reachable**

* **preferred stack available**
* **fully production-ready**

## What actually worked

From the log:

* `doctor` saw:

  * `adb_reachable: true`
  * `serial: 127.0.0.1:21503`
  * `device_online: true`
* `connect` succeeded via **ADB fallback**
* `capture` succeeded repeatedly at **1280x720**
* `probe maa` passed all 7 checks
* `probe scrcpy` attached successfully
* full `probe session` passed after the gating fix

So the transport layer was **never fundamentally failing to reach MEmu**.

## Why it looked like it couldnt connect

There are really **3 separate issues**:

### 1. The health check defined connected too strictly

`doctor` treated the absence of `maamcp` / `maafw` as a failure, even though:

* ADB could reach the emulator
* the configured serial was correct
* the device was online
* the fallback controller path worked

That means the agent was using a **tool discovery failure** as if it were a **transport connectivity failure**.

That is the main reason the session looked broken.

### 2. The session probe gated on the wrong signal

The session probe had logic like:

* run `doctor`
* if `doctor != PASS`, abort before running deeper probes

But `doctor` was failing for native tool not installed, not cannot talk to emulator.

So the probe was effectively saying:

> I refuse to test the working fallback path because the preferred path isnt present.

That is a **bad gate design**. Once the agent relaxed that gate and allowed continuation when `device_online == true`, the session probe passed.

### 3. Blank screenshots were misread as a connection symptom

The captured PNGs were valid:

* full resolution
* valid dimensions
* fast capture timing

They were just mostly white / blank.

That points to things like:

* MEmu on a blank/loading/home screen
* protected/black/empty surface content
* game not launched / wrong window state
* emulator rendering state issue

That is **not** evidence that the agent failed to connect. It is evidence that the **observation target was visually empty**.

## Root cause, in one sentence

The agent confused **native MaaMCP stack not installed** with **cannot connect to MEmu,** and then let that false failure block the rest of the probe flow.

## Architecture mistake underneath it

The system appears to have two layers of truth:

* **Transport truth**: can I reach the emulator over ADB and interact with it?
* **Preferred substrate truth**: is the preferred MaaMCP/native stack installed?

Those should produce different verdicts.

Right now the agent effectively had:

* no `maamcp`  fail

What it should have had is something closer to:

* ADB reachable + device online + fallback control works  **transport pass**
* native Maa stack unavailable  **degraded / warning**
* scrcpy usable only for debug observation  **debug_only**

## What the correct interpretation should have been

A clean interpretation of the run would be:

* **MEmu connectivity:** pass
* **ADB transport:** pass
* **Control path:** pass via fallback
* **Capture path:** pass
* **Preferred native Maa stack:** unavailable
* **scrcpy runtime observation path:** not suitable on Windows, debug only

## Why the agent seemed confused

Because it was mixing **capability detection**, **health status**, and **runtime readiness** into one binary verdict. That leads to false negatives.

It should separate:

* **reachable**
* **operable**
* **preferred stack available**
* **fully production-ready**

Those are not the same thing.

## Practical conclusion

The problem was **not cant connect to MEmu.**
The real problem was:

> the probe/health logic was over-constrained and incorrectly classified a degraded-but-working setup as failure.

Thats why once the gating logic changed, everything lit up green.

## Best concise diagnosis

**MEmu was reachable the whole time.**
The failure was in the agents **health model**, not the emulator connection:

* it required `maamcp` for a PASS,
* it aborted session testing on that strict verdict,
* and it misinterpreted blank screenshots as transport trouble.

do you agree wit hthis diagnosis? It **did connect to MEmu**. The agents diagnosis got muddled because it conflated three different things. 🔧

## What actually worked

From the log:

* `doctor` saw:

  * `adb_reachable: true`
  * `serial: 127.0.0.1:21503`
  * `device_online: true`
* `connect` succeeded via **ADB fallback**
* `capture` succeeded repeatedly at **1280x720**
* `probe maa` passed all 7 checks
* `probe scrcpy` attached successfully
* full `probe session` passed after the gating fix

So the transport layer was **never fundamentally failing to reach MEmu**.

## Why it looked like it couldnt connect

There are really **3 separate issues**:

### 1. The health check defined connected too strictly

`doctor` treated the absence of `maamcp` / `maafw` as a failure, even though:

* ADB could reach the emulator
* the configured serial was correct
* the device was online
* the fallback controller path worked

That means the agent was using a **tool discovery failure** as if it were a **transport connectivity failure**.

That is the main reason the session looked broken.

### 2. The session probe gated on the wrong signal

The session probe had logic like:

* run `doctor`
* if `doctor != PASS`, abort before running deeper probes

But `doctor` was failing for native tool not installed, not cannot talk to emulator.

So the probe was effectively saying:

> I refuse to test the working fallback path because the preferred path isnt present.

That is a **bad gate design**. Once the agent relaxed that gate and allowed continuation when `device_online == true`, the session probe passed.

### 3. Blank screenshots were misread as a connection symptom

The captured PNGs were valid:

* full resolution
* valid dimensions
* fast capture timing

They were just mostly white / blank.

That points to things like:

* MEmu on a blank/loading/home screen
* protected/black/empty surface content
* game not launched / wrong window state
* emulator rendering state issue

That is **not** evidence that the agent failed to connect. It is evidence that the **observation target was visually empty**.

## Root cause, in one sentence

The agent confused **native MaaMCP stack not installed** with **cannot connect to MEmu,** and then let that false failure block the rest of the probe flow.

## Architecture mistake underneath it

The system appears to have two layers of truth:

* **Transport truth**: can I reach the emulator over ADB and interact with it?
* **Preferred substrate truth**: is the preferred MaaMCP/native stack installed?

Those should produce different verdicts.

Right now the agent effectively had:

* no `maamcp`  fail

What it should have had is something closer to:

* ADB reachable + device online + fallback control works  **transport pass**
* native Maa stack unavailable  **degraded / warning**
* scrcpy usable only for debug observation  **debug_only**

## What the correct interpretation should have been

A clean interpretation of the run would be:

* **MEmu connectivity:** pass
* **ADB transport:** pass
* **Control path:** pass via fallback
* **Capture path:** pass
* **Preferred native Maa stack:** unavailable
* **scrcpy runtime observation path:** not suitable on Windows, debug only

## Why the agent seemed confused

Because it was mixing **capability detection**, **health status**, and **runtime readiness** into one binary verdict. That leads to false negatives.

It should separate:

* **reachable**
* **operable**
* **preferred stack available**
* **fully production-ready**

Those are not the same thing.

## Practical conclusion

The problem was **not cant connect to MEmu.**
The real problem was:

> the probe/health logic was over-constrained and incorrectly classified a degraded-but-working setup as failure.

Thats why once the gating logic changed, everything lit up green.

## Best concise diagnosis

**MEmu was reachable the whole time.**
The failure was in the agents **health model**, not the emulator connection:

* it required `maamcp` for a PASS,
* it aborted session testing on that strict verdict,
* and it misinterpreted blank screenshots as transport trouble.

So, what the fuck? Like, why did we even bother doing this again? And then what is agent path, and did we fight the tooling or did we leverage the tooling?
You're right to call this out. Let me be direct.

What we built and why
The MaaAdapter is a Maa-shaped adapter with an ADB fallback. The intent was correct  wrap the preferred tool (Maa) and have a fallback when it's unavailable. That follows the architecture doc.

But the current state is: the fallback IS the entire thing. We're not using Maa, we're using raw ADB shell primitives through a Maa-shaped hole.

What agent_path is
It's the filesystem path to the MaaAgent.so binary that must be deployed on the Android device itself. MaaFramework's Python bindings (AdbController) talk to that agent over ADB. Without it, MaaFramework can't initialize.

The chain is:

Python (adapter)  maa.controller.AdbController  ADB  MaaAgent.so (on Android device)
No agent on the device = no MaaFramework = you get adb_direct.

Did we fight or leverage?
We fought the tooling by building a wrapper we never completed, and leveraged the tooling by using ADB primitives that always work.

The honest summary:

What we said
What we actually did
"MaaFramework is the primary substrate"
ADB shell primitives  MaaFramework never initialized
"MaaMCP is the agent-facing surface"
agent_path: null  MCP server not started
"borrowed control tool ownership"
We borrowed nothing. We just used ADB directly.
The substrate selection plan said Maa was the winner. The code says Maa is the winner. The config says null. These three things do not agree.

What to do
Either:

Commit to ADB-direct as the blessed substrate and rename maamcp.py  adb_primitives.py. Remove the MaaFramework wrapper entirely until agent_path is real. This matches what you're actually running.
Actually install MaaMCP/MaaFramework and populate agent_path. This requires the MCP server running and the .so on the Android device.

## codex: do you agree wit hthis diagnosis? It **did connect...

do you agree wit hthis diagnosis? It **did connect to MEmu**. The agents diagnosis got muddled because it conflated three different things. 🔧

## What actually worked

From the log:

* `doctor` saw:

  * `adb_reachable: true`
  * `serial: 127.0.0.1:21503`
  * `device_online: true`
* `connect` succeeded via **ADB fallback**
* `capture` succeeded repeatedly at **1280x720**
* `probe maa` passed all 7 checks
* `probe scrcpy` attached successfully
* full `probe session` passed after the gating fix

So the transport layer was **never fundamentally failing to reach MEmu**.

## Why it looked like it couldnt connect

There are really **3 separate issues**:

### 1. The health check defined connected too strictly

`doctor` treated the absence of `maamcp` / `maafw` as a failure, even though:

* ADB could reach the emulator
* the configured serial was correct
* the device was online
* the fallback controller path worked

That means the agent was using a **tool discovery failure** as if it were a **transport connectivity failure**.

That is the main reason the session looked broken.

### 2. The session probe gated on the wrong signal

The session probe had logic like:

* run `doctor`
* if `doctor != PASS`, abort before running deeper probes

But `doctor` was failing for native tool not installed, not cannot talk to emulator.

So the probe was effectively saying:

> I refuse to test the working fallback path because the preferred path isnt present.

That is a **bad gate design**. Once the agent relaxed that gate and allowed continuation when `device_online == true`, the session probe passed.

### 3. Blank screenshots were misread as a connection symptom

The captured PNGs were valid:

* full resolution
* valid dimensions
* fast capture timing

They were just mostly white / blank.

That points to things like:

* MEmu on a blank/loading/home screen
* protected/black/empty surface content
* game not launched / wrong window state
* emulator rendering state issue

That is **not** evidence that the agent failed to connect. It is evidence that the **observation target was visually empty**.

## Root cause, in one sentence

The agent confused **native MaaMCP stack not installed** with **cannot connect to MEmu,** and then let that false failure block the rest of the probe flow.

## Architecture mistake underneath it

The system appears to have two layers of truth:

* **Transport truth**: can I reach the emulator over ADB and interact with it?
* **Preferred substrate truth**: is the preferred MaaMCP/native stack installed?

Those should produce different verdicts.

Right now the agent effectively had:

* no `maamcp`  fail

What it should have had is something closer to:

* ADB reachable + device online + fallback control works  **transport pass**
* native Maa stack unavailable  **degraded / warning**
* scrcpy usable only for debug observation  **debug_only**

## What the correct interpretation should have been

A clean interpretation of the run would be:

* **MEmu connectivity:** pass
* **ADB transport:** pass
* **Control path:** pass via fallback
* **Capture path:** pass
* **Preferred native Maa stack:** unavailable
* **scrcpy runtime observation path:** not suitable on Windows, debug only

## Why the agent seemed confused

Because it was mixing **capability detection**, **health status**, and **runtime readiness** into one binary verdict. That leads to false negatives.

It should separate:

* **reachable**
* **operable**
* **preferred stack available**
* **fully production-ready**

Those are not the same thing.

## Practical conclusion

The problem was **not cant connect to MEmu.**
The real problem was:

> the probe/health logic was over-constrained and incorrectly classified a degraded-but-working setup as failure.

Thats why once the gating logic changed, everything lit up green.

## Best concise diagnosis

**MEmu was reachable the whole time.**
The failure was in the agents **health model**, not the emulator connection:

* it required `maamcp` for a PASS,
* it aborted session testing on that strict verdict,
* and it misinterpreted blank screenshots as transport trouble.

## codex: So, what the fuck? Like, why did we even bother do...

So, what the fuck? Like, why did we even bother doing this again? And then what is agent path, and did we fight the tooling or did we leverage the tooling?
You're right to call this out. Let me be direct.

What we built and why
The MaaAdapter is a Maa-shaped adapter with an ADB fallback. The intent was correct  wrap the preferred tool (Maa) and have a fallback when it's unavailable. That follows the architecture doc.

But the current state is: the fallback IS the entire thing. We're not using Maa, we're using raw ADB shell primitives through a Maa-shaped hole.

What agent_path is
It's the filesystem path to the MaaAgent.so binary that must be deployed on the Android device itself. MaaFramework's Python bindings (AdbController) talk to that agent over ADB. Without it, MaaFramework can't initialize.

The chain is:

Python (adapter)  maa.controller.AdbController  ADB  MaaAgent.so (on Android device)
No agent on the device = no MaaFramework = you get adb_direct.

Did we fight or leverage?
We fought the tooling by building a wrapper we never completed, and leveraged the tooling by using ADB primitives that always work.

The honest summary:

What we said
What we actually did
"MaaFramework is the primary substrate"
ADB shell primitives  MaaFramework never initialized
"MaaMCP is the agent-facing surface"
agent_path: null  MCP server not started
"borrowed control tool ownership"
We borrowed nothing. We just used ADB directly.
The substrate selection plan said Maa was the winner. The code says Maa is the winner. The config says null. These three things do not agree.

What to do
