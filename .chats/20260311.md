# Pass 3: Models & Runtimes (Decisions 1-13)

### 1. Primary Orchestrator Model
*   **Decision**: Gemini 3.1 Pro via API for Levels 0-2 (NSA, CEO, Director, Coordinator).
*   **The "WHY"**: Orchestration requires immense context retention (1-2 million tokens) to read `portfolio.md` across 15 repos simultaneously without dropping context. 
*   **Discarded Alternative (Comparison)**: Claude 3.5 Sonnet (200k limit) was discarded for the top layer. While Sonnet is superior at atomic coding, it forces artificial server-side compaction (which drops details) when ingesting massive repository states, unlike Gemini 3.1 Pro's native long-context ingest.
*   **Implications**: Forces us to use Google's Vertex or AI Studio API keys. Top-level orchestration will be slightly slower but significantly more contextually aware.

### 2. Orchestrator Runtime Framework
*   **Decision**: Pure Python 3.14 via Cron jobs and stateless execution scripts.
*   **The "WHY"**: Python allows strict, predictable, standard file I/O operations without framework locking.
*   **Discarded Alternative (Comparison)**: Temporal Workflows and LangGraph. Temporal is the 2026 industry standard for durable AI execution, but it requires deploying a massive server and worker nodes. For a single-user Personal AI, this is immense overkill. We discarded it because `git commit` to a local directory acts as a sufficient durable database.
*   **Implications**: Crash recovery must be handled via pure Git state checks. No automatic GUI for monitoring workflow execution graphs.

### 3. Execution Worker Model
*   **Decision**: Claude 3.5 Sonnet (and newer 4/4.5 Sonnet betas) for Level 3 Workers.
*   **The "WHY"**: Anthropic models definitively lead 2026 benchmarks for AST-aware file editing, tool calling precision, and CLI interaction.
*   **Discarded Alternative (Comparison)**: Gemini 3.1 Pro and Llama 3. Gemini occasionally prioritizes safety filters over strict code enforcement in long contexts.
*   **Implications**: Binds our execution layer to Anthropic's ecosystem.

### 4. Execution Runtime Engine
*   **Decision**: `claude-code` CLI wrapped as a Python Subprocess.
*   **The "WHY"**: Avoid replacing a perfectly engineered wheel. Anthropic spent millions building a secure "Think -> Act -> Observe" master loop into their CLI. 
*   **Discarded Alternative (Comparison)**: OpenClaw's custom Node.js execution loop. OpenClaw relies entirely on its own engine, which was hit by massive path-traversal and remote-execution vulnerabilities in early 2026.
*   **Implications**: The Coordinator Python script treats the Worker as a black box. It passes inputs via STDIN, waits for process exit, and evaluates the `git diff`.

### 5. Fallback Coding Model
*   **Decision**: GitHub Copilot API / Codex.
*   **The "WHY"**: Massive reasoning tokens (Sonnet/Pro) are too expensive to generate a basic regex or boilerplate CSS class.
*   **Discarded Alternative (Comparison)**: Using Claude Sonnet for every single keystroke. Discarded due to 50-hour cycle budget exhaustion.
*   **Implications**: Requires Coordinator to classify tasks into `[TRIVIAL]` vs `[COMPLEX]` prior to dispatch.

### 6. Background Compaction Model
*   **Decision**: Local open-weight models (Llama 3 / Mistral) via Ollama.
*   **The "WHY"**: 50 hours of execution logs produce gigabytes of text. API-based summarization costs are unjustifiable.
*   **Discarded Alternative (Comparison)**: Manus relies on server-side API summarization to shrink the context window. 
*   **Implications**: The host machine must have sufficient VRAM to run Ollama nightly to compact `log.md` into embeddings.

### 7. Vision / Multimodal Validation Model
*   **Decision**: Gemini 3.1 Pro Vision.
*   **The "WHY"**: When a frontend Worker finishes layout edits, the Coordinator takes a Playwright screenshot and uses Gemini Vision to verify if the "button is overlapping the header."
*   **Discarded Alternative (Comparison)**: DOM-only scraping (BeautifulSoup). Discarded because CSS rules often create visual overlaps that are perfectly valid in the DOM hierarchy.
*   **Implications**: Adds a visual QA step to the workflow pipeline, increasing token cost but drastically reducing broken UI pushes.

### 8. Memory Retrieval Embedder
*   **Decision**: Specialized local embedding models (e.g., Nomic-Embed-Text via Ollama).
*   **The "WHY"**: Fast, private, and free vectorization of historical architecture decisions.
*   **Discarded Alternative (Comparison)**: OpenAI `ada-002` embeddings. Discarded to prevent leaking private repo architecture rules over the wire.
*   **Implications**: Embeddings remain consistent and completely offline.

### 9. Orchestration Hosting & Lifecycles
*   **Decision**: Local Daemon / Wait-State execution. No permanent active listeners.
*   **The "WHY"**: Infinite loops cost API money if an agent gets stuck. The CEO wakes up, does triage, writes to `portfolio.md`, and terminates.
*   **Discarded Alternative (Comparison)**: OpenClaw's 24/7 WebSocket Gateway. Discarded because exposing an always-on listener on `0.0.0.0` led to CVE-2026-25253 remote takeovers. 
*   **Implications**: The system is naturally rate-limited by its wake-up cadence (cron).

### 10. Worker Environment Isolation
*   **Decision**: Ephemeral Docker Containers.
*   **The "WHY"**: AI agents write destructive code (`rm -rf /`).
*   **Discarded Alternative (Comparison)**: Running `claude-code` directly on the host OS. Discarded as catastrophically unsafe for autonomous overnight runs.
*   **Implications**: Slows down worker spin-up time (Docker boot latency), but guarantees host OS safety.

### 11. State Machine Strategy
*   **Decision**: Pure stateless markdown reading. The state *is* the file directory.
*   **The "WHY"**: Markdown checklists (`[x]`) are parsed easily by Gemini and are human-readable.
*   **Discarded Alternative (Comparison)**: Manus's masked token logits (enforcing state via raw LLM probability manipulation). Discarded because we cannot mask logits over the public API; we must enforce state via physical IO loops.
*   **Implications**: Slower transitions between states, as files must be read/written to disk at each step.

### 12. Parallelism Model
*   **Decision**: Single-threaded Orchestrator, Multi-threaded Workers.
*   **The "WHY"**: The Director manages 1 roadmap sequentially. It can dispatch 3 Workers to 3 distinct branches parallelly, but it only evaluates their PRs sequentially.
*   **Discarded Alternative (Comparison)**: Claude Code Agent Teams (sub-agents talking directly to each other). Discarded because inter-agent chatter is impossible to systematically monitor or rollback.
*   **Implications**: Strict bottleneck at the PR review phase.

### 13. Dependency Injection
*   **Decision**: Explicit module passing in Python. No heavy frameworks (No LangChain DI).
*   **The "WHY"**: Maximizes script readability and debuggability. 
*   **Discarded Alternative (Comparison)**: LangChain/LlamaIndex. Discarded because their hidden abstractions obscure the actual prompts sent to the LLM, making debugging complex prompt-jailbreaks impossible.
*   **Implications**: Python scripts will be slightly longer but 100% transparent.

# Pass 3: Tools & Skills (Decisions 14-25)

### 14. Core Tooling Standard
*   **Decision**: 100% enforcement of the Model Context Protocol (MCP). 
*   **The "WHY"**: Standardizes the interface between Gemini (Orchestration) and Claude (Execution) using universal tool schemas.
*   **Discarded Alternative (Comparison)**: Custom JSON-schema API function definitions. Discarded because updating custom tools for different models creates immense tech debt. 
*   **Implications**: Any custom skill we write must be packaged as an MCP server.

### 15. Tools Provisioning Architecture
*   **Decision**: Just-In-Time (JIT) Mounting. Workers boot naked.
*   **The "WHY"**: Giving an agent AWS access when it is only tasked with centering a CSS div is a massive security risk.
*   **Discarded Alternative (Comparison)**: OpenClaw's default capability pooling. OpenClaw exposes all global skills, making the agent a high-value target for prompt injection.
*   **Implications**: The Coordinator must perform a Gemini API call *before* launching the Worker to answer: "What MCP servers are needed for this task?"

### 16. Web Browsing Approach
*   **Decision**: Playwright MCP over standard Playwright usage.
*   **The "WHY"**: Playwright MCP passes structured DOM state to the LLM, drastically reducing context window bloat compared to passing raw HTML strings.
*   **Discarded Alternative (Comparison)**: Raw vision-based browser steering (taking screenshots and predicting X,Y coordinates). Discarded because it is highly flaky on responsive designs.
*   **Implications**: Requires running a Node.js Playwright MCP server sidecar in the frontend Worker container.

### 17. Web Scraping & Research Approach
*   **Decision**: Firecrawl / Jina AI APIs over pure Playwright scraping.
*   **The "WHY"**: When the NSA or CEO needs to research external documentation, LLM-optimized scraping APIs convert convoluted JS-heavy sites directly to Markdown.
*   **Discarded Alternative (Comparison)**: Attempting to use the Playwright MCP to "read" docs. Discarded because navigating multi-page documentation sites via agent loops wastes too many API tokens.
*   **Implications**: Adds dependency on external scraping APIs for the Research/Orchestration tier.

### 18. Terminal Execution Boundary
*   **Decision**: Restricted MCP Command Runner.
*   **The "WHY"**: Prevents the LLM from executing `curl malicious-script | bash`.
*   **Discarded Alternative (Comparison)**: Claude-Code's native `/bin/bash` tool. While powerful, allowing untrusted generative models raw terminal access is hazardous. We utilize Claude Code's server-based sandboxing concepts but implement a local whitelist.
*   **Implications**: The Worker can only execute commands whitelisted by the Coordinator (e.g., `npm run test`, `git status`).

### 19. Skill Definition & Storage
*   **Decision**: Strongly typed Python/Node MCP interfaces in `/skills/`.
*   **The "WHY"**: Defends against Skill Injection attacks.
*   **Discarded Alternative (Comparison)**: OpenClaw's `skill.md` natural language definitions. Thousands of OpenClaw instances were compromised in 2026 because anyone could write malicious prompt instructions into a markdown skill file.
*   **Implications**: Writing new skills requires actual coding, not just prompt engineering.

### 20. File System Access Topology
*   **Decision**: The host repo is mounted `Read-Only`. A specific `.tmp_sandbox` branch is mounted `Read-Write`.
*   **The "WHY"**: Physical containment. The agent cannot modify the host's `main` branch codebase.
*   **Discarded Alternative (Comparison)**: Standard Claude-Code local execution, which edits files in place. Discarded to prevent accidental destruction of uncommitted work.
*   **Implications**: Output must be `git push`ed from the container and evaluated as a Pull Request on the host.

### 21. Third-Party API Skill Proxying
*   **Decision**: Local MCP Proxy Server to handle AWS/DB queries.
*   **The "WHY"**: Prevents sending physical Database Passwords or AWS Tokens into Anthropic's cloud context window. The agent asks the local Proxy to execute a query, and the Proxy holds the keys.
*   **Discarded Alternative (Comparison)**: Direct token injection via `.env` into the LLM prompt. Discarded for obvious security compliance reasons (MCP Confused Deputy Problem mitigations).
*   **Implications**: Complex tool setup to ensure zero-trust secret handling.

### 22. IDE Tooling Capability
*   **Decision**: MCP bridge to LSP (Language Server Protocol).
*   **The "WHY"**: Allows the agent to query "Find all references to `auth()`. instead of using brute-force `grep`.
*   **Discarded Alternative (Comparison)**: Standard `grep_search`. LLMs are notoriously bad at regex. LSP provides deterministic compiler-level syntax trees to the agent.
*   **Implications**: Requires spinning up a Python/Node LSP server in the background of the Worker container.

### 23. Identity Guardrails
*   **Decision**: Pre-injected `agent.md` identity file.
*   **The "WHY"**: An Infrastructure Agent scaling AWS should not suddenly decide to redesign the CSS.
*   **Discarded Alternative (Comparison)**: Self-prompting agents. Discarded because agents suffer persona drift over 50 hours.
*   **Implications**: `agent.md` is appended to the `system` prompt block invariably by Claude-Code on boot.

### 24. Tool Execution Approval
*   **Decision**: The Worker *requests* execution; the MCP server executes.
*   **The "WHY"**: Maintains the security perimeter. The LLM dictates intent, the structural code evaluates path traversal flaws before clicking "Go".
*   **Discarded Alternative (Comparison)**: Giving the LLM native OS functions.
*   **Implications**: Development time overhead to write defensive MCP servers.

### 25. Secret Management
*   **Decision**: Explicit ENV injections mapping to host `uuid` secrets via Docker daemon.
*   **The "WHY"**: Ensures logs generated by the agent never contain hardcoded tokens. 
*   **Discarded Alternative (Comparison)**: Hardcoded `.env` files in the repo root. Discarded due to risk of agent inadvertently running `cat .env` and sending it to the LLM API.
*   **Implications**: Secrets are managed securely by the host Python script.

# Pass 3: Memory & Resource Storage (Decisions 26-38)

### 26. Primary State Database
*   **Decision**: Git is the only database for operational state.
*   **The "WHY"**: Version control is mathematically perfect for tracking state changes, rollbacks, and concurrent branches.
*   **Discarded Alternative (Comparison)**: Complex Postgres/MongoDB state tables used by traditional AI platforms. Discarded because an invisible DB state prevents a human from seamlessly taking over if the AI gets stuck. Git state is inherently inspectable.
*   **Implications**: Python Orchestrators rely heavily on parsing `git status` and `git diff`.

### 27. Knowledge Base Storage
*   **Decision**: Plain text Markdown in `/docs/`.
*   **The "WHY"**: Markdown is native to humans and the absolute best format for LLM parsing token-for-token.
*   **Discarded Alternative (Comparison)**: Confluence/Notion API integrations. Discarded to eliminate network latency and external dependencies from the core loop.
*   **Implications**: The repo acts as a monorepo containing both its code and its entire organizational brain.

### 28. Vector Database Protocol
*   **Decision**: Local `sqlite-vec`.
*   **The "WHY"**: Long-horizon AI needs to retrieve past architectural decisions ("Why Prisma?"). `sqlite-vec` runs inside a `.sqlite` file requiring zero server infrastructure.
*   **Discarded Alternative (Comparison)**: Pinecone or Milvus. Discarded because cloud vector DBs introduce network latency, cost, and complexity completely unnecessary for single-repo context management.
*   **Implications**: Vector memory travels portably alongside the codebase.

### 29. Transient Worker Memory
*   **Decision**: Wiped strictly upon Docker container exit.
*   **The "WHY"**: Forces the Worker to rely on the codebase, not its conversational cache, guaranteeing reproducible results.
*   **Discarded Alternative (Comparison)**: Preserving the chat history between Worker loops. Discarded due to "context rot" (hallucinations resulting from stale prompt data).
*   **Implications**: The Worker starts "stupid" every single tick, forcing explicit `plan.md` instructions.

### 30. Orchestrator Context Compaction
*   **Decision**: Wipe the Gemini conversational array every 5 Director loops.
*   **The "WHY"**: The Director gets confused tracking endless "done/failed/done" messages.
*   **Discarded Alternative (Comparison)**: Claude models' automatic server-side compaction. Discarded because Anthropic decides what to compress. We want deterministic control. We wipe the memory and substitute an explicit 2-sentence `status.md` summary.
*   **Implications**: The Director loses the "vibe" of the conversation but definitively retains the hard data.

### 31. Memory Retrieval Protocol
*   **Decision**: A dedicated Level 2 "Memory Retrieval Agent" exposed via MCP tool.
*   **The "WHY"**: Replaces the standard RAG pipeline. If the Worker needs to know the company CSS colors, it queries the Memory Agent, which intelligently scans `/docs` and `sqlite-vec` to return a concise answer, rather than the Worker blindly using `grep`.
*   **Discarded Alternative (Comparison)**: Standard "blind" RAG (top K thresholding). Discarded because basic semantic search often returns irrelevant blocks of code, polluting the prompt.
*   **Implications**: Requires parallel orchestration of a specific search-optimized agent prompt.

### 32. Checkpoint Serialization
*   **Decision**: `[ ]`, `[/]`, and `[x]` markdown checkboxes in `plan.md`.
*   **The "WHY"**: It forces the LLM to output structured intent without the fragility of massive JSON blobs.
*   **Discarded Alternative (Comparison)**: JSON State Machines. Discarded because LLMs frequently break JSON formatting when asked to interleave natural language thought processes.
*   **Implications**: The Python parser must rely on regex to evaluate task completion status.

### 33. Embedding Strategy
*   **Decision**: Local Nomic Embed Text via Ollama.
*   **The "WHY"**: High speed, offline, dense embeddings for markdown.
*   **Discarded Alternative (Comparison)**: `text-embedding-3-small`. Discarded to maintain offline independence.
*   **Implications**: Requires initial setup of local models, but embeds instantly.

### 34. Architecture Decision Records (ADR)
*   **Decision**: Mandated explicit `.md` files in `docs/adr/`.
*   **The "WHY"**: Forces the NSA/CEO to codify *why* a decision was made before execution, serving as the highest-weight context for the Memory Agent.
*   **Discarded Alternative (Comparison)**: Implicit memory stored in chat logs. Discarded as untrackable.
*   **Implications**: The workflow must pause execution to draft an ADR for major shifts.

### 35. Daily Logging
*   **Decision**: Append-only `log.md` files rolled over daily.
*   **The "WHY"**: Mimics OpenClaw's ephemeral memory tracking. Provides a chronological audit trail for the human.
*   **Discarded Alternative (Comparison)**: Standard terminal stdout logs. Discarded because they are inaccessible to the LLM.
*   **Implications**: Disk I/O overhead on every action.

### 36. Resource Caching
*   **Decision**: Scraping Docs once and storing in `/docs/resources/cached/`.
*   **The "WHY"**: Prevents the agent from downloading Next.js API docs from the internet 50 times an hour.
*   **Discarded Alternative (Comparison)**: Manus's "live search" approach. Live searches burn tokens and network time.
*   **Implications**: Requires a manual or semi-automated cache invalidation strategy.

### 37. Diff Tracking Context
*   **Decision**: Native `git diff HEAD` is the ultimate success metric fed back to the Coordinator.
*   **The "WHY"**: The LLM's summary of what it did is often a hallucination. The raw git diff is mathematically true.
*   **Discarded Alternative (Comparison)**: Relying on Claude-Code's stdout "Task Complete" message.
*   **Implications**: Context windows must accommodate unified git diff formats.

### 38. Cost Tracking Memory
*   **Decision**: SQLite ledger updated post-API request.
*   **The "WHY"**: Hard budget enforcement.
*   **Discarded Alternative (Comparison)**: Relying on provider dashboards. Discarded as they lag behind real-time execution.
*   **Implications**: Orchestrator calculates token counts using `tiktoken` equivalent before dispatch.

# Pass 3: Workflows & Hierarchy (Decisions 39-50)

### 39. The Orchestrator Hierarchy
*   **Decision**: 5-Level Nested Execution (North Star Advisor -> CEO -> Director -> Coordinator -> Worker).
*   **The "WHY"**: 50 hours of execution without hierarchical scoping ends in context collapse. An agent working on a React component does not need to know the entire CEO's portfolio strategy.
*   **Discarded Alternative (Comparison)**: Claude Code "Agent Teams" (a mesh network of agents passing messages). Discarded because an unstructured mesh network quickly devolves into infinite API loops during edge cases.
*   **Implications**: Workflow transitions are rigid and vertical. The Coordinator cannot talk to the CEO.

### 40. Task Atomicity
*   **Decision**: Workers own strictly 1 `[ ]` checklist item and exit immediately upon completion.
*   **The "WHY"**: LLMs degrade sharply when attempting to juggle 5 different file modifications in one stream.
*   **Discarded Alternative (Comparison)**: OpenClaw's continuous execution session. Discarded because if OpenClaw gets stuck on step 4, the session hangs. Our Workers exit after 1 step, whether successful or failed.
*   **Implications**: The Coordinator must run `git diff` after every single Worker exit before spinning up the next Worker container.

### 41. Inter-Agent Communication Protocol
*   **Decision**: Markdown File I/O exclusively. No internal REST/gRPC.
*   **The "WHY"**: APIs fail silently. Markdown files can be read by humans mid-run.
*   **Discarded Alternative (Comparison)**: Temporal Workflows passing JSON state objects. Discarded to maintain 100% human-inspectability at every node.
*   **Implications**: High Disk I/O. State parsing relies on deterministic regex inside Python.

### 42. Strategic Validation Firewall
*   **Decision**: The North Star Advisor script is physically isolated from the codebase directory.
*   **The "WHY"**: Agents reading compiler errors routinely hallucinate that they need to rewrite the entire architecture to fix a syntax bug.
*   **Discarded Alternative (Comparison)**: Appending the `northstar.md` straight into the Worker's `system` prompt. Discarded because the Worker will ignore it under the pressure of immediate coding tasks.
*   **Implications**: Strategic drift is caught at the Director level, preventing the Worker from initiating massive rewrites.

### 43. Deployment / Merge Strategy
*   **Decision**: Strict Human-in-the-Loop PR Reviews via the Daily Briefing.
*   **The "WHY"**: A 50-hour autonomous agent run *will* make at least one destructive decision. Autonomous merging to `main` is disallowed.
*   **Discarded Alternative (Comparison)**: Full CI/CD Autonomous Deployment. Discarded to prevent the AI from accidentally dropping production databases overnight.
*   **Implications**: Output velocity is bounded by the human's ability to review PRs each morning.

### 44. Failure Handling Mechanics
*   **Decision**: 3-Strikes Circuit Breaker per atomic task.
*   **The "WHY"**: "Context Rot" occurs when an agent tries to fix the same error 10 times. Setting a hard limit saves thousands in API costs.
*   **Discarded Alternative (Comparison)**: OpenClaw's indefinite retry loop. Discarded as financially ruined.
*   **Implications**: Tasks marked `[BLOCKED]` must be manually resolved by the human, or the CEO must pivot.

### 45. Version Control Paradigm
*   **Decision**: Feature Branches only (`feat/agent-auth`). No commits to `main` by Python/LLM.
*   **The "WHY"**: Total containment of generated blast radius.
*   **Discarded Alternative (Comparison)**: Conventional Trunk-based development with live pushes. Discarded because the agent's code is mathematically untrusted.
*   **Implications**: The repo will contain many abandoned or failed AI branches that need pruning.

### 46. Global Kill Switch
*   **Decision**: `OVERRIDE.md` file watcher runs on a separate thread in every Python script.
*   **The "WHY"**: If an agent starts a destructive behavior loop, the user needs a 0-millisecond kill command.
*   **Discarded Alternative (Comparison)**: Relying on `Ctrl+C` in a hidden Docker container. Discarded because finding the right PID to kill takes too long.
*   **Implications**: Writing the word "STOP" into `OVERRIDE.md` cleanly exits all Python scripts immediately.

### 47. Automated Testing Enforceability
*   **Decision**: Coordinator strictly requires `npm run test` (or equivalent) to pass before verifying an atomic task `[x]`.
*   **The "WHY"**: Diff outputs can look correct but break the build.
*   **Discarded Alternative (Comparison)**: Assuming the code works if the LLM says "Task Complete".
*   **Implications**: The agent will burn additional API tokens writing tests and fixing them before it is allowed to exit the Docker container.

### 48. Safe Triage Protocol (Overnight Shift)
*   **Decision**: The CEO only dispatches isolated, non-breaking modifications to the sandboxes overnight.
*   **The "WHY"**: Core schema migrations should not happen while the user is asleep.
*   **Discarded Alternative (Comparison)**: Equal weighting of all tasks. 
*   **Implications**: The CEO must semantically classify the human's `todo.txt` inputs into `[SAFE]` vs `[RISKY]`.

### 49. The Morning Briefing 
*   **Decision**: The CEO script compiles `daily_briefing.md` summarizing completed PRs, costs, and BLOCKED tasks.
*   **The "WHY"**: The human needs a 2-minute summary of the last 14 hours of execution.
*   **Discarded Alternative (Comparison)**: Reading raw stdout logs. Discarded as unintelligible to humans.
*   **Implications**: The CEO must ingest the summaries of all active Directors at 7:00 AM.

### 50. Budget Enforcement 
*   **Decision**: Hard pre-flight cost checks calculated locally via `tiktoken` by the Infra Director before ANY orchestration dispatch.
*   **The "WHY"**: Gemini 3.1 Pro 2M-token API calls can quickly burn hundreds of dollars if a loop malfunctions.
*   **Discarded Alternative (Comparison)**: Relying on cloud provider delayed billing alerts. Discarded because they often lag 24 hours behind actual spend.
*   **Implications**: Any job exceeding the hardcoded $X token cost ceiling is immediately rejected back to the user.
