# Polypipeline Project Briefing — March 2026

## Problem Statement

A house in Hendersonville, TN has a half-height basement/crawlspace with a poorly documented plumbing system assembled across multiple renovation rounds. The system cannot be understood by visual inspection alone. The goal is to produce a trustworthy node-based connectivity graph of the plumbing, with metric spatial coordinates.

## What We Have

### Scan Sources

Two independent 3D capture tracks from an iPhone with LiDAR:

**Polycam (subscription expired, no new captures possible):**
- 24 capture directories, 13 with usable geometry
- Only ONE confirmed crawlspace scan: `6C6BF087` (6,249 frames, 15×3×20m, pier-block grid visible)
- The crawlspace is where the pipes physically are
- Several house interior/exterior scans
- Known issues: one capture (`7160ce32`) has 100x inflated coordinate scale; raw mesh exporter only reads first mesh from multi-mesh glTF files
- Existing corpus of 13 PLY exports already on disk

**RTAB-Map (active capture tool, 16 databases total, 12 export-worthy):**

|---|---|---|---|---|---|---|
| front2.db | 3.7 GB | 2,370 | 10m | 12.3×9.7×2.1 | **228** | house_interior |
| 260316-143108.db | 1.6 GB | 2,352 | 10m | 13.8×12.1×4.0 | **220** | house_interior |
| 260315-151736.db | 2.2 GB | 1,079 | 4m | 15.5×14.2×1.6 | **92** | house_interior |

Key fact: **No RTAB-Map databases contain crawlspace data.** Crawlspace = Polycam only. RTAB-Map = house interior + outdoor plumbing only.

### Available Tooling

- `rtabmap-export.exe` (v0.23.1, local): exports cloud (`--cloud`) or mesh (`--mesh`) from .db files. Key params: `--opt` (0=full optimization, 1=iterative, 2=pre-computed poses, 3=raw odometry), `--voxel`, `--ba` (bundle adjustment)
- `rtabmap-reprocess.exe`: can reprocess single DBs or merge multiple DBs (`"a.db;b.db" merged.db`)
- Python scripts: `compare_ply.py` (bbox/centroid comparison), `icp_align_mesh.py` (alignment), various inventory scripts
- `uv` for Python execution
- Remote machine "modal" on Tailscale: 16 cores, 128GB RAM, Linux, no GPU, no rtabmap installed

### What's Been Done So Far

**Completed exports (investigation runs):**
- P-B01: Polycam `3410c293` raw mesh → 225K vertices, 7MB
- P-B02: Polycam `7160ce32` raw mesh → wrong scale (100x inflated)
- R-B01: `pumpfast.db` direct cloud export → 572K points, 17MB
- R-B02: `260302-064248.db` direct cloud export → 2.4M points, 72MB
- R-B03 series: `pumpfast.db` reprocess-then-export experiments → proved that reprocess/export combos are calibration targets; naive defaults produced garbage (2.5K points or wrong-scale output); `--opt 3` after reprocess stayed close to baseline
- R-NEW01: `front2.db` direct cloud export → **3.2M points**, 96MB
- R-NEW02: `260316-143108.db` direct cloud export → **3.6M points**, 107MB
- R-NEW03: `260315-151736.db` direct cloud export → **1.7M points**, 49MB

**Key empirical findings (lessons_learned.md):**
1. FPFH/RANSAC feature matching fails completely on this geometry (walls, floors lack distinctive features)
2. ICP threshold must be ≤0.05m; 2m threshold produces fake high-fitness results
3. Fitness reflects overlap percentage, not alignment quality (low fitness + good RMSE = different areas, correctly aligned)
4. Compass heading (±2-3° actual precision) is useful for initial rotation alignment
5. Same-heading Polycam captures are ~9cm apart (ARKit is cm-accurate)
6. iPhone LiDAR CAN resolve pipes as geometry at 0.3-2m range
7. Gaussian splatting is wrong output format for pipe mapping (view-dependent, not geometry)
8. RTAB-Map reprocess is not automatically an improvement — export mode after reprocess matters critically
9. The ceiling scan (260315-213744-recording) has zero loop closures and a 3.6m coordinate jump

### What We Don't Know

1. **What area each RTAB-Map scan covers** — we have extent dimensions but no room-level identification for most databases. Without this, we can't determine which scans overlap.
2. **Whether RTAB-Map multi-session merge works on this data** — only tested reprocess on the small `pumpfast.db` (4 loops). Never tried on the big 200+ loop databases or cross-session merge.
3. **Whether the Polycam crawlspace scan can be aligned to the RTAB-Map interior model** — this is essential since crawlspace data exists only in Polycam.
4. **The actual geometric quality of the new baseline exports** — we have point counts and file sizes but haven't validated internal consistency or compared overlapping regions.
5. **Which processing path produces the most accurate geometry** — the investigation plan lists many experiment matrix entries still pending.

## The Fundamental Question

What is the most efficient path from "16 RTAB-Map databases + 13 Polycam captures on disk" to "a unified, geometrically accurate 3D model of the house (interior + crawlspace) suitable for tracing plumbing"?

### Candidate Approaches

**A: Native RTAB-Map multi-session merge**
- Use `rtabmap-reprocess "a.db;b.db;..." merged.db` to merge interior databases at the SLAM level
- Exploits visual features and loop closure detection across sessions
- Works in the native rich representation, not lossy exported clouds
- Risk: R-B03 showed reprocess can produce garbage on small datasets; unknown behavior on large multi-session merge
- Would still need separate Polycam→RTAB-Map alignment for crawlspace

**B: Export-then-align with ICP**
- Export best cloud from each database independently
- Use ICP (with compass heading initialization where available) to register pairs
- More control over each step; failure modes are visible
- Risk: ICP on featureless geometry is known to be hard (FPFH already failed)
- Requires identifying overlapping pairs first

**C: Hybrid — merge RTAB-Map databases where possible, ICP for cross-track alignment**
- Group RTAB-Map databases by likely overlap (need to identify rooms first)
- Merge within groups using native reprocess
- Export merged groups, then ICP across groups and with Polycam
- Balances native SLAM quality with practical control

**D: Focus on the crawlspace first**
- The pipes are in the crawlspace (6C6BF087 Polycam scan)
- Interior scans provide context (sink/toilet locations) but the actual pipe geometry is underground
- Could extract and segment the crawlspace scan first, then use interior scans only for validation/context
- Risk: one scan may not cover the entire crawlspace

**E: Something else entirely**
- Are there better tools for multi-session alignment? (CloudCompare, Open3D global registration, etc.)
- Should we be using the mesh export (`--mesh`) instead of cloud for alignment?
- Is there value in the Polycam interior scans for areas the RTAB-Map didn't cover well?
- Should we install rtabmap on the remote 128GB machine for parallel processing?

## Constraints

- No new Polycam captures possible (subscription expired)
- New RTAB-Map captures are possible
- FPFH/RANSAC is banned for this geometry (empirically failed)
- ICP threshold must be ≤0.05m
- Processing should use `uv` for Python, native CLI for RTAB-Map
- Must keep Polycam and RTAB-Map as separate source tracks

2. What should we do FIRST?
3. What are we probably overlooking?

## What I Now Know Is Real

**The exact syntax for reprocessing with different parameters:**

To convert your database taken from iPhone to use SIFT features: `rtabmap-reprocess --Kp/DetectorStrategy 1 --Vis/FeatureType 1 rtabmap.db output.db`

That's matlabbe himself, in a GitHub discussion. Every `--Parameter/Name value` flag overrides the default for that reprocessing run.

**What reprocessing can and cannot change:**

Changing Core parameters will affect only the processing functions: refining the links, adding new loop closures, or how the map is optimized. If we replay the database with different parameters, they will only affect the mapping step, not odometry.

This is critical. **You cannot redo odometry.** The ARKit poses baked into the .db are fixed. What you CAN redo is everything the SLAM back-end does: feature extraction, vocabulary building, loop closure detection, loop closure verification, graph optimization. That's where all the quality lives for your use case.

**To redo odometry, you need raw recording mode data:**

If you want to re-run odometry with different parameters with the same data, you will have to record data at higher frame rate.

So for Mapping Session .db files: poses are locked, SLAM is redoable. For Data Recording .db files (60Hz raw): everything is redoable including odometry.

## The Actual Parameters That Matter

From the wiki, the source code, and matlabbe's forum responses, here's what you can override on `rtabmap-reprocess`:

| Parameter | What it controls | Default (iOS) | Aggressive desktop value | Why |
|---|---|---|---|---|
| `Kp/DetectorStrategy` | Feature detector type | 6 (GFTT-BRIEF) | 1 (SIFT) or 0 (SURF) | SIFT/SURF are far more discriminative for loop closure in large or repetitive environments; too slow for real-time on phone, trivial on desktop |
| `Vis/FeatureType` | Feature type for visual loop closure verification | 6 (GFTT-BRIEF) | 1 (SIFT) or 0 (SURF) | Same reason; must match Kp/DetectorStrategy |
| `Kp/MaxFeatures` | Max features per image for vocabulary | 400 | 2000+ | More features = richer vocabulary = better loop closure discrimination |
| `Vis/MaxFeatures` | Max features for loop closure transform estimation | 400 | 1000+ | More correspondences for verifying loop closures |
| `Rtabmap/LoopThr` | Loop closure acceptance threshold | 0.11 | 0.05-0.08 | Lower = accepts weaker hypotheses; phone is conservative, desktop can afford more and reject bad ones via Max Optimization Error |
| `Vis/MinInliers` | Minimum inliers to accept loop closure transform | 25 | 15-20 | Lower = accepts loop closures with fewer geometric matches |
| `RGBD/OptimizeMaxError` | Reject loop closures causing errors above this factor | 2x | 3x | Slightly more permissive; lets marginal loop closures through if they don't blow up the graph |

## What I Now Know Is Real

**The exact syntax for reprocessing with different parameters:**

To convert your database taken from iPhone to use SIFT features: `rtabmap-reprocess --Kp/DetectorStrategy 1 --Vis/FeatureType 1 rtabmap.db output.db`

That's matlabbe himself, in a GitHub discussion. Every `--Parameter/Name value` flag overrides the default for that reprocessing run.

**What reprocessing can and cannot change:**

Changing Core parameters will affect only the processing functions: refining the links, adding new loop closures, or how the map is optimized. If we replay the database with different parameters, they will only affect the mapping step, not odometry.

This is critical. **You cannot redo odometry.** The ARKit poses baked into the .db are fixed. What you CAN redo is everything the SLAM back-end does: feature extraction, vocabulary building, loop closure detection, loop closure verification, graph optimization. That's where all the quality lives for your use case.

**To redo odometry, you need raw recording mode data:**

If you want to re-run odometry with different parameters with the same data, you will have to record data at higher frame rate.

So for Mapping Session .db files: poses are locked, SLAM is redoable. For Data Recording .db files (60Hz raw): everything is redoable including odometry.
